## §五十 Bug 修复 — Session 列表轮询闪烁

**Phase**: 57 Hotfix
**背景**: §四十八~§四十九 引入的轮询机制导致 session 列表每 10 秒闪烁一次。

**根因分析**:

| # | 问题 | 位置 | 影响 |
|---|------|------|------|
| 1 | `useRunningSessions` 每次轮询无条件 `setRunningKeys(new Set())`，即使内容未变也创建新引用 | `useRunningSessions.ts` | SessionList 每 10s 全部重渲染 |
| 2 | `useSubagentStatus` 每次轮询无条件 `setStatusMap(new Map())`，同理 | `useSubagentStatus.ts` | SessionList 每 5s 全部重渲染 |
| 3 | `useRunningSessions` 变化时调 `fetchSessions()` → `sessionStore` 无条件设 `loading: true` → `Sidebar.tsx` 根据 loading 隐藏整个 SessionList → 重新加载后显示 | `sessionStore.ts` + `Sidebar.tsx` | 列表先消失再出现 = 闪烁 |

**修复方案**:

1. **`useRunningSessions.ts`**: 将 `setRunningKeys()` 移到 changed 判断内部，仅在内容真正变化时才更新 state
2. **`useSubagentStatus.ts`**: 新增 `mapsEqual()` 深比较函数（JSON 序列化对比），仅在内容变化时 setState
3. **`sessionStore.ts`**: `fetchSessions()` 区分首次加载（设 loading）和后台刷新（静默更新，不设 loading）
4. **`Sidebar.tsx`**: 移除 `loading` 门控，SessionList 始终渲染（数据更新由 zustand 驱动，无需隐藏/显示切换）

**额外修复 — 前端未重新构建**:
- 66cd02a 提交修改了源码但未执行 `npm run build`，`frontend/dist/` 仍为旧版本
- 需重新 `npm run build` + 重启 webserver 才能生效

**验收**:
1. session 列表在任务执行期间不再闪烁
2. 运行中标识（绿点）和 subagent 进度仍正常显示
3. 新 session 出现时列表仍能自动刷新

## §五十五 并发 task 时用户消息丢失修复

**Phase**: 59
**来源**: Plan B R2 验收 G2

**背景**: 当 session 空闲时用户发消息与 cron 消息同时到达，先到的 cron 通过 `_run_task_sdk()` 创建 task 后，用户的请求到达 `_handle_execute_stream()` 检测到 task 已 running，直接 `_attach_to_existing_task(existing); return` — 用户消息被完全丢弃。

**根因代码**: `worker.py` `_handle_execute_stream()` 中检测到已有 running task 时，仅 attach SSE 而未保留用户消息。

**改动**:
- `worker.py` `_handle_execute_stream()`: 检测到 task 已 running 时，先将用户消息通过 `_inject_queue.put({"role": "user", "content": prefixed})` 注入运行中的 task，再 attach SSE
- inject 格式与 `SessionMessenger.send_to_session()` / `WorkerCronExecutor.send_to_session()` 保持一致（dict 格式 `{"role": "user", "content": ...}`）

**验收**:
1. 用户发消息时若 session 恰好有 cron task running，用户消息被 inject 到 task 中
2. SSE 正常 attach，用户可看到后续输出
3. 不影响正常（无并发）的消息发送流程

## §五十六 session 从 running 变为 idle 时前端最终刷新

**Phase**: 59
**来源**: Plan B R2 验收 G6

**背景**: 主 session spawn subagent 后自己停止（非 running，无 SSE），subagent 完成后结果通过 inject 激活主 session。主 session 被激活后只回复一句话就结束，前端没有 SSE 连接，5 秒轮询可能来不及捕获这次短暂的回复。

**根因**: task 由后端 inject 触发（通过 `SessionMessenger` 或 `_run_task_sdk`），前端从未发起请求，所以没有 SSE 连接。task 从 running 到 done 可能只有几秒，5 秒轮询可能刚好错过。

**改动 1 — MessageList.tsx 最终刷新**:
- 新增 `wasRunningRef` 追踪上一次 running 状态
- auto-refresh useEffect 中检测 `wasRunning && !isRunning` 状态变化，执行一次 `loadMessages()` 最终刷新

**改动 2 — messageStore.ts refreshMessages**:
- 新增 `refreshMessages()` 方法：获取消息但不先清空（preserves component state / expand-collapse）
- auto-refresh 轮询改用 `refreshMessages` 替代 `loadMessages`，避免刷新时 UI 闪烁

**改动 3 — MessageItem.tsx 展开状态持久化**:
- 新增 module-level `_expandedIds` Set 和 `usePersistedExpand` hook
- `ToolCallLine`、`ToolProcessCollapsible`、`SystemInjectCard` 的展开/折叠状态跨刷新持久化
- 配合 `refreshMessages` 实现自动刷新时不丢失用户的展开操作

**验收**:
1. 主 session 被 subagent 返回激活后，即使回复很快结束，前端也能捕获到最终消息
2. 自动刷新期间展开的工具调用详情不会被折叠
3. 自动刷新不会导致消息列表闪烁

## §五十七 自动刷新时保持滚动位置

**来源**: Plan D 验收 — 用户反馈 subagent session 自动刷新时页面自动滚到底部

**问题**: subagent session 的 auto-refresh 5s 轮询会强制把页面滚到底部，用户无法向上浏览历史。

**根因**（两层）:
1. **scroll 事件监听器从未绑定**: `useEffect([], [])` 在组件首次 render 时执行，但此时 MessageList 处于 loading/empty early-return 状态，`scrollContainerRef.current` 为 null，导致 scroll/wheel/pointer 事件监听器从未被 attach。`userScrolledAwayRef` 永远保持初始值 `false`。
2. **subagent session 无 SSE/attach**: subagent task 不在 worker `_tasks` 中（由 parent task 内部 spawn，不经 `/execute`），`checkRunningTask` 的 `fetchTaskStatus` 返回 `unknown`，不设 `sending=true`。auto-refresh interval 从 session 打开起就在运行，但因 flag 永远为 false，每次 tick 都执行 refresh + scrollIntoView。

**修复 — MessageList.tsx**:
1. **callback ref + useState**: 用 `scrollContainerCallbackRef` 替代直接 `ref={scrollContainerRef}`，通过 `useState` 追踪 DOM 节点可用性，useEffect 依赖 `[scrollContainer]`，确保事件监听器在 early-return 条件消除后正确绑定
2. **userScrolledAwayRef 粘性标志**: 用户滚离底部（>300px）时锁定为 true，只在用户主动交互（wheel/touch/pointer）回到底部时重置为 false，不受 DOM 高度变化影响
3. **userInteractingRef 交互检测**: 通过 wheel/touchstart/pointerdown 事件标记用户主动操作，区分用户滚动 vs 程序/布局触发的 scroll 事件
4. **auto-refresh interval**: 检查 `userScrolledAwayRef` 而非 `isNearBottom()`，用户滚离时完全跳过 refresh（零 re-render = 零干扰），标记 `pendingRefreshRef` 待用户滚回底部后补刷

**验收**:
1. 用户在底部时，auto-refresh 正常刷新并跟随新消息 ✅
2. 用户向上滚动浏览历史时，auto-refresh 完全跳过，零闪烁零干扰 ✅
3. 用户从上方滚回底部时，立即触发 pending refresh 获取最新内容 ✅
4. running→idle 最终刷新尊重 userScrolledAwayRef，不强制滚到底部 ✅
